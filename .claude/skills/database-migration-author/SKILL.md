# Database Migration Author — GCE Data Platform

Use this skill when adding or reviewing a new database migration. The stack is
**EF Core 8 for migration orchestration only** — runtime data access is Dapper. Every
migration executes raw SQL scripts embedded as assembly resources; there are no EF
entity mappings.

Three recurring bug classes this skill exists to prevent:

1. **Non-idempotent migrations** that crash when re-applied against a database that
   already has the object.
2. **Baseline drift** — a migration adds a column but `baseline_create.sql` still shows
   the old shape, so a fresh `drop-all + baseline` install is out of sync with a
   migrated install.
3. **Seed data breakage** — a new NOT NULL column without a DEFAULT makes
   `seed_test_data.sql` fail on re-run.

## File set for every migration

Each migration needs **five** edits / creations:

```
backend/GcePlatform.Api/Migrations/
  YYYYMMDDNNNNNN_YourMigrationName.cs          # EF class
  YYYYMMDDNNNNNN_YourMigrationName.Designer.cs # auto-generated twin
  Scripts/YourMigrationName.Up.sql             # forward
  Scripts/YourMigrationName.Down.sql           # rollback

backend/GcePlatform.Api/GcePlatform.Api.csproj # 2 new <EmbeddedResource> lines
database/ddl/tables/baseline_create.sql        # keep in sync (see §Sync rule)
database/scripts/drop-all.sql                  # if you introduced new droppable objects
database/dml/seed_test_data.sql                # if the change affects seed rows
```

### `.cs` class template

```csharp
// File: Migrations/YYYYMMDDNNNNNN_YourMigrationName.cs
using System.Text.RegularExpressions;
using Microsoft.EntityFrameworkCore.Migrations;

#nullable disable

namespace GcePlatform.Api.Migrations
{
    /// <inheritdoc />
    public partial class YourMigrationName : Migration
    {
        /// <inheritdoc />
        /// One-sentence description of what this migration does and why.
        protected override void Up(MigrationBuilder migrationBuilder)
        {
            ExecuteSqlScript(migrationBuilder,
                "GcePlatform.Api.Migrations.Scripts.YourMigrationName.Up.sql");
        }

        /// <inheritdoc />
        protected override void Down(MigrationBuilder migrationBuilder)
        {
            ExecuteSqlScript(migrationBuilder,
                "GcePlatform.Api.Migrations.Scripts.YourMigrationName.Down.sql");
        }

        private static void ExecuteSqlScript(MigrationBuilder migrationBuilder, string resourceName)
        {
            var assembly = typeof(YourMigrationName).Assembly;
            using var stream = assembly.GetManifestResourceStream(resourceName)
                ?? throw new InvalidOperationException(
                    $"Embedded SQL resource '{resourceName}' was not found.");

            using var reader = new StreamReader(stream);
            var sql = reader.ReadToEnd();

            var batches = Regex.Split(
                sql,
                @"^\s*GO\s*$",
                RegexOptions.Multiline | RegexOptions.IgnoreCase);

            foreach (var batch in batches)
            {
                var trimmed = batch.Trim();
                if (!string.IsNullOrWhiteSpace(trimmed))
                    migrationBuilder.Sql(trimmed, suppressTransaction: true);
            }
        }
    }
}
```

Copy verbatim from the most recent migration (e.g.
`20260424000001_SplitKpiPermissions.cs`) and rename. The `ExecuteSqlScript` helper is
the same in every file — do not simplify or share it across migrations.

### `.Designer.cs` template

```csharp
// <auto-generated />
using GcePlatform.Api.Data;
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Infrastructure;
using Microsoft.EntityFrameworkCore.Migrations;

#nullable disable

namespace GcePlatform.Api.Migrations
{
    [DbContext(typeof(PlatformDbContext))]
    [Migration("YYYYMMDDNNNNNN_YourMigrationName")]
    partial class YourMigrationName
    {
        /// <inheritdoc />
        protected override void BuildTargetModel(ModelBuilder modelBuilder)
        {
#pragma warning disable 612, 618
            modelBuilder
                .HasAnnotation("ProductVersion", "8.0.11")
                .HasAnnotation("Relational:MaxIdentifierLength", 128);

            SqlServerModelBuilderExtensions.UseIdentityColumns(modelBuilder);
#pragma warning restore 612, 618
        }
    }
}
```

`BuildTargetModel` body is intentionally empty — the schema is managed entirely by the
SQL scripts, so there are no entities to track. The `[Migration(...)]` attribute string
MUST match the filename (without `.cs`) exactly.

### csproj — EmbeddedResource

Append under the existing `<ItemGroup>` around `Migrations\Scripts\...` entries:

```xml
<!-- YourMigrationName: one-line description -->
<EmbeddedResource Include="Migrations\Scripts\YourMigrationName.Up.sql" />
<EmbeddedResource Include="Migrations\Scripts\YourMigrationName.Down.sql" />
```

Both Up and Down MUST be embedded. The resource name looked up at runtime is
`GcePlatform.Api.Migrations.Scripts.YourMigrationName.Up.sql` (default namespace +
path-as-dots + filename).

### Migration ID format

`YYYYMMDDNNNNNN_ClassName` — date plus a 6-digit sequence. Use the next free sequence
for the day (`20260424000001`, `20260424000002`). The date is UTC-ish; the sequence
lets multiple migrations land on the same day without collision.

## SQL Up / Down — idempotency is mandatory

Every structural statement MUST be guarded. SQL Server doesn't have `CREATE TABLE IF
NOT EXISTS`, so we use explicit metadata checks.

### Tables

```sql
IF OBJECT_ID('Schema.TableName', 'U') IS NULL
BEGIN
    CREATE TABLE Schema.TableName ( ... );
END;
GO
```

### Columns

```sql
IF COL_LENGTH('Schema.TableName', 'NewColumn') IS NULL
BEGIN
    ALTER TABLE Schema.TableName ADD NewColumn INT NULL;
END;
GO
```

### Views / procedures / functions

```sql
-- CREATE OR ALTER is naturally idempotent
CREATE OR ALTER VIEW App.vThings AS
    SELECT ... ;
GO
```

### Indexes

```sql
IF NOT EXISTS (
    SELECT 1 FROM sys.indexes
    WHERE object_id = OBJECT_ID('Schema.TableName')
      AND name = 'IX_Thing_Something'
)
    CREATE INDEX IX_Thing_Something ON Schema.TableName (SomeColumn);
GO
```

### Default constraints

```sql
IF NOT EXISTS (
    SELECT 1 FROM sys.default_constraints
    WHERE parent_object_id = OBJECT_ID('Schema.TableName')
      AND name = 'DF_Thing_Modified'
)
    ALTER TABLE Schema.TableName
        ADD CONSTRAINT DF_Thing_Modified DEFAULT SYSUTCDATETIME() FOR ModifiedOnUtc;
GO
```

### Column nullability change

Backfill NULLs BEFORE tightening:

```sql
UPDATE Schema.TableName
   SET ModifiedOnUtc = CreatedOnUtc
   WHERE ModifiedOnUtc IS NULL;

IF EXISTS (
    SELECT 1 FROM sys.columns
    WHERE object_id = OBJECT_ID('Schema.TableName')
      AND name = 'ModifiedOnUtc'
      AND is_nullable = 1
)
    ALTER TABLE Schema.TableName ALTER COLUMN ModifiedOnUtc DATETIME2(3) NOT NULL;
GO
```

`ALTER COLUMN` doesn't support inline `DEFAULT`; add the `DF_...` constraint in a
separate guarded block.

### Seed / lookup rows

```sql
IF NOT EXISTS (SELECT 1 FROM App.PlatformPermission WHERE PermissionCode = 'kpi.admin')
    INSERT INTO App.PlatformPermission (PermissionCode, DisplayName, Description, Category, SortOrder)
    VALUES ('kpi.admin', 'Manage KPI Library', '...', 'KPI', 50);
GO
```

### Hard-delete

```sql
-- Remove dependent rows first (FK safety)
DELETE rp
FROM App.PlatformRolePermission AS rp
JOIN App.PlatformPermission AS p ON p.PermissionId = rp.PermissionId
WHERE p.PermissionCode = 'kpi.manage';
GO

IF EXISTS (SELECT 1 FROM App.PlatformPermission WHERE PermissionCode = 'kpi.manage')
    DELETE FROM App.PlatformPermission WHERE PermissionCode = 'kpi.manage';
GO
```

### `GO` separators

- Batch-separate every DDL statement with `GO` on its own line.
- The C# `ExecuteSqlScript` helper splits on `^\s*GO\s*$` (multiline, case-insensitive).
- `suppressTransaction: true` is passed per-batch — which is why some DDL (e.g. `CREATE
  INDEX` on a large table) works even though it can't run in a transaction.

### Down must undo Up

The `Down.sql` script mirrors the Up in reverse order (indexes before defaults before
nullability before columns before tables). Always provide a real Down — it's the only
way `dotnet ef database update <previous>` recovers from a bad forward migration in
dev.

Acceptable exception: irreversible data operations (e.g. deleting a legacy lookup row
that had in-use FKs). In that case, write a Down that restores the minimum shape but
document in the SQL comment block that data is not recoverable.

## Naming conventions

| Object | Pattern | Example |
|---|---|---|
| Primary key | `PK_<Table>` | `PK_Tag` |
| Foreign key | `FK_<Child>_<Parent>` | `FK_KpiTemplate_Definition` |
| Unique index | `UX_<Table>_<Scope>` | `UX_Tag_Name` |
| Non-unique index | `IX_<Table>_<Scope>` | `IX_KpiSub_LockedBy_Principal` |
| Check constraint | `CK_<Table>_<Rule>` | `CK_KpiSub_LockState` |
| Default constraint | `DF_<Table>_<Column>` | `DF_Tag_Modified` |
| View | `v<CamelCase>` in `App` / `Sec` schema | `App.vAccounts` |
| Power BI view | `vw_PBI_<Name>` in `Reporting` | `Reporting.vw_PBI_AccountKPI` |
| Stored procedure | `usp_<Action><Subject>` or `<Verb><Subject>` | `App.usp_UpsertKpiAssignmentTemplate` |

Abbreviations used in existing code (`KpiSub`, `KpiTemplate`, `AccRolePolicy`) are
shortened to keep names under 128 chars. Follow the adjacent convention — don't invent
a new abbreviation scheme.

## Audit columns — the canonical 4

Every new row-tracked table gets these four columns, ALL NOT NULL with defaults:

```sql
CreatedOnUtc   DATETIME2(3)  NOT NULL CONSTRAINT DF_<Table>_CreatedOn  DEFAULT SYSUTCDATETIME(),
CreatedBy      NVARCHAR(128) NOT NULL CONSTRAINT DF_<Table>_CreatedBy  DEFAULT SESSION_USER,
ModifiedOnUtc  DATETIME2(3)  NOT NULL CONSTRAINT DF_<Table>_ModifiedOn DEFAULT SYSUTCDATETIME(),
ModifiedBy     NVARCHAR(128) NOT NULL CONSTRAINT DF_<Table>_ModifiedBy DEFAULT SESSION_USER
```

`NVARCHAR(128)` matches SQL Server's identifier length and `SESSION_USER`. Never use
`NVARCHAR(200)` or `NVARCHAR(MAX)` for `CreatedBy` / `ModifiedBy`.

## The sync rule — baseline + drop-all + seed

This is a durable project rule. Every migration MUST be accompanied by corresponding
updates to the three sister files so that:

- `drop-all.sql` + `baseline_create.sql` → gives the same shape as
- `baseline_create.sql` (at release N) + migrations 1..N → which gives

```
drop-all  →  baseline  →  migrations up to now  ==  baseline (at this commit)
```

Checklist for the sync rule:

- [ ] If the migration creates a new table/view/procedure/function, add the matching
      `CREATE` to `baseline_create.sql` AND the matching `DROP IF EXISTS` to
      `drop-all.sql`.
- [ ] If the migration alters a column, update the column definition inside its
      `CREATE TABLE` block in `baseline_create.sql`.
- [ ] If the migration adds a seed row (e.g. a new permission code), add the `IF NOT
      EXISTS INSERT` block to the corresponding seed region in `baseline_create.sql`.
- [ ] If the migration drops a seed row, remove the `IF NOT EXISTS INSERT` block too.
- [ ] If the migration introduces a new type of row that `seed_test_data.sql` should
      exercise (so the feature is testable out of the box), add rows to the seed.
- [ ] If the migration tightens a NOT NULL, confirm `seed_test_data.sql` still
      generates valid rows (usually OK because seeds call SPs that rely on defaults).

If you are unsure whether a change belongs in `baseline_create.sql`: it does. The
baseline is the source of truth for "what a fresh DB looks like at this commit".

## Verification steps (always run)

```bash
# From backend/GcePlatform.Api/
dotnet build                     # 0 warnings, 0 errors (pre-existing DbConnectionFactory warning is OK)
dotnet ef migrations list        # your migration appears with (Pending)
dotnet ef database update        # applies cleanly to dev DB
dotnet ef database update <previous>  # rolls back cleanly via Down.sql
dotnet ef database update        # rolls forward again (idempotency check)
```

Any migration that fails the roll-back / roll-forward cycle is broken.

## Review Checklist

### File set
- [ ] `.cs` + `.Designer.cs` both present, class name matches filename
- [ ] Up.sql + Down.sql both present under `Migrations/Scripts/`
- [ ] Both SQL files added to `GcePlatform.Api.csproj` as `<EmbeddedResource>` with a
      comment line describing what the migration does
- [ ] Migration ID format `YYYYMMDDNNNNNN_PascalCaseName`
- [ ] `[Migration("...")]` attribute string matches the filename exactly

### Idempotency
- [ ] `CREATE TABLE` guarded with `IF OBJECT_ID('...', 'U') IS NULL`
- [ ] `ALTER TABLE ADD` guarded with `IF COL_LENGTH('...', '...') IS NULL`
- [ ] `CREATE INDEX` guarded with `IF NOT EXISTS (SELECT 1 FROM sys.indexes WHERE ...)`
- [ ] Default constraints guarded with `IF NOT EXISTS (SELECT 1 FROM sys.default_constraints WHERE ...)`
- [ ] `ALTER COLUMN ... NOT NULL` preceded by a NULL-backfill `UPDATE`
- [ ] Views / procedures / functions use `CREATE OR ALTER`
- [ ] Seed `INSERT` guarded with `IF NOT EXISTS (SELECT 1 FROM ...)`
- [ ] `DELETE` FK-dependent rows before dropping the parent

### GO batching
- [ ] Every DDL statement separated by `GO` on its own line
- [ ] No T-SQL DDL inside a single un-GO'd block (e.g. don't `IF NOT EXISTS CREATE TABLE`
      followed immediately by `CREATE INDEX` without a `GO`)

### Down script
- [ ] Mirrors Up in reverse order
- [ ] Uses the same idempotency guards
- [ ] Drops only what the Up created — doesn't touch siblings

### Audit columns (on any new table)
- [ ] `CreatedOnUtc DATETIME2(3) NOT NULL DEFAULT SYSUTCDATETIME()`
- [ ] `CreatedBy NVARCHAR(128) NOT NULL DEFAULT SESSION_USER`
- [ ] `ModifiedOnUtc DATETIME2(3) NOT NULL DEFAULT SYSUTCDATETIME()`
- [ ] `ModifiedBy NVARCHAR(128) NOT NULL DEFAULT SESSION_USER`
- [ ] DEFAULTs named `DF_<Table>_<Column>` so later migrations can drop/rebuild them

### Naming
- [ ] Index / FK / default constraint names follow the `UX_` / `IX_` / `FK_` / `DF_` conventions
- [ ] Stored procedures: `usp_<Action><Subject>` under `App` schema
- [ ] Views: `v<Name>` under `App` / `Sec` / `Reporting`

### Baseline + drop-all + seed sync
- [ ] `baseline_create.sql` reflects the post-migration shape (column list, indexes, seed rows, view body)
- [ ] `drop-all.sql` drops any new table / view / function / procedure
- [ ] `seed_test_data.sql` still runs clean after a `drop-all → baseline → seed` cycle
- [ ] If the migration introduces a testable new feature, `seed_test_data.sql` exercises
      it (e.g. a new tag type, a new permission code, a new assignment group)

### Rollback
- [ ] `dotnet ef database update <previous-migration>` succeeds
- [ ] `dotnet ef database update` (roll forward again) succeeds
- [ ] Irreversible data changes are explicitly called out in the Down script's comment block

## Common anti-patterns to grep for

| Pattern | Why it's a red flag |
|---|---|
| `CREATE TABLE` without `IF OBJECT_ID` guard | Non-idempotent; crashes on re-run |
| `ALTER TABLE ... ADD <col>` without `COL_LENGTH` guard | Same |
| `CREATE INDEX` without `sys.indexes NOT EXISTS` guard | Same |
| `ALTER COLUMN ... NOT NULL` without a preceding backfill UPDATE | Migration crashes if NULL rows exist |
| New table without the 4 audit columns | Untraceable writes |
| `NVARCHAR(MAX)` or `NVARCHAR(200)` for `CreatedBy` / `ModifiedBy` | Wrong width; should be `NVARCHAR(128)` |
| New seed row (`INSERT INTO App.PlatformPermission ...`) missing from `baseline_create.sql` | Baseline drift |
| New table missing from `drop-all.sql` | Clean-install cycles fail |
| Down.sql missing or empty | Can't recover from a bad forward migration in dev |
| Migration ID without the date prefix (`_SplitKpiPermissions.cs` alone) | Breaks ordering |
| `BuildTargetModel` body that actually maps entities | This repo manages schema entirely via SQL — keep it empty |
